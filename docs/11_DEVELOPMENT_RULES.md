# 11_DEVELOPMENT_RULES.md

Version: V1.0  
System: HR Daily AI

---

# 1. Purpose

This document defines mandatory development rules for all implementation in HR Daily AI V1.

These rules are STRICT and NON-NEGOTIABLE.

All code generated by Codex or any developer MUST comply.

---

# 2. Core Principle

The system is designed for:

- AI-assisted development
- Predictable architecture
- Maintainability
- Minimal complexity
- Single-developer scalability

---

# 3. Architectural Rules

## 3.1 Layer Enforcement

All code MUST follow:

```text
UI -> Feature -> Service -> Repository -> Prisma -> Database
```

## 3.2 Forbidden Architecture Violations

The following are STRICTLY forbidden:

- Direct Prisma usage in UI
- Business logic inside API routes
- AI calls inside UI components
- AI calls inside API routes directly
- Cross-feature direct coupling
- Unstructured data storage for core features

## 3.3 Feature Isolation Rule

Each feature MUST:

- Be self-contained
- Not depend on internal structure of other features
- Communicate only via services or shared types

---

# 4. Coding Rules

## 4.1 TypeScript Rules

- TypeScript strict mode MUST be enabled
- NO `any` type allowed
- All exported functions MUST have explicit return types

## 4.2 Code Quality Rules

- No duplicated logic
- No unused code
- No commented-out dead code
- No inline business logic in UI components

## 4.3 Naming Conventions

- camelCase for variables and functions
- PascalCase for components and models
- kebab-case for files only when required by framework
- Consistent domain naming (log, review, planner, knowledge)

## 4.4 File Rules

Each file MUST:

- Have a single responsibility
- Be under 300 lines (recommended)
- Be placed in correct layer folder

---

# 5. Service Layer Rules

## 5.1 Mandatory Usage

All business logic MUST be in:

```text
src/services/
```

## 5.2 Responsibilities

Service layer handles:

- Business logic
- Data aggregation
- AI orchestration
- Cross-repository coordination

## 5.3 Forbidden in Service Layer

- Direct UI rendering
- Direct Prisma queries (must go through repository)
- Hardcoded prompts
- UI state management

---

# 6. Repository Layer Rules

## 6.1 Responsibilities

Repository layer ONLY handles:

- CRUD operations
- Database queries
- Data persistence

## 6.2 Forbidden in Repository Layer

- Business logic
- AI logic
- Data transformation beyond persistence mapping

---

# 7. AI Rules

## 7.1 AI Isolation Rule

AI MUST only exist in:

```text
src/ai/
```

## 7.2 AI Call Restrictions

- AI MUST NOT be called from UI
- AI MUST NOT be called from API routes directly
- AI MUST be called via Service Layer only

## 7.3 AI Output Rule

- MUST be JSON only
- MUST follow schema
- MUST be validated before saving
- MUST NOT be used directly without parsing
- MUST store trace fields required by database design

---

# 8. API Rules

## 8.1 Thin Controller Rule

API routes MUST:

- Receive request
- Validate input
- Call service layer
- Return response

No business logic allowed.

## 8.2 Response Format

All APIs MUST follow:

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

---

# 9. Database Rules

## 9.1 Prisma Only Rule

- All database access MUST use Prisma
- No raw SQL allowed in V1

## 9.2 Migration Rule

- All schema changes MUST use Prisma migrations
- Manual DB changes are forbidden

## 9.3 Data Integrity Rules

- UUID as primary key
- No soft delete in V1
- No historical versioning
- All timestamps in UTC
- Numeric recruiting count fields must be greater than or equal to 0

---

# 10. AI Prompt Rules

## 10.1 Externalization Rule

All prompts MUST be stored in:

```text
/prompts/
```

## 10.2 No Inline Prompt Rule

- NO prompts inside code files
- NO hardcoded prompt content inside business logic
- Prompt Builder may inject structured input into Markdown prompt templates

## 10.3 Determinism Rule

Prompts MUST:

- Produce stable outputs for same input
- Avoid randomness
- Avoid hallucination

---

# 11. Performance Rules

- Non-AI API response < 300ms
- AI response target < 10s
- No blocking UI operations
- AI calls must not freeze UI
- AI APIs must handle timeout explicitly

---

# 12. Security Rules

- No API keys in frontend
- All secrets via environment variables
- `.env` must never be committed
- No logging of sensitive credentials
- No prompt leakage to client

---

# 13. Git Rules

## 13.1 Commit Discipline

- One task = one commit
- No mixed-feature commits
- No large undocumented commits
- Every completed task must be independently commit-ready

## 13.2 Branch Strategy

- main -> stable
- feature/* -> development

---

# 14. Docker Rules

- All services MUST run in Docker
- No dependency on local machine runtime (except Docker)
- One command startup required

Canonical startup command from repository root:

```bash
docker compose -f docker/docker-compose.yml up --build
```

Short form is only allowed if a root-level compose file is added later:

```bash
docker compose up --build
```

---

# 15. Testing Rules (V1 Minimal)

Required checks:

- Basic build test
- Type check
- Lint check

Commands:

```bash
npm run build
npm run typecheck
npm run lint
```

No full test suite required in V1.

---

# 16. Out of Scope (V1)

- Microservices
- Event-driven architecture
- CQRS
- DDD full implementation
- Complex design patterns (Factory/Strategy unless necessary)
- Full test coverage (unit/integration/e2e)

---

# 17. Summary

These rules ensure:

- Predictable AI-generated code
- Stable architecture
- Minimal debugging cost
- High maintainability
- Safe evolution beyond V1
